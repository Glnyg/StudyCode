# 开工前设计冻结清单

## 这篇解决什么问题
- 这篇笔记回答一个很关键的问题：
  - 我们现在还缺不缺“大架构”？
  - 还是已经该停止摇摆，进入详细设计和开工？

## 先说结论
- 现在这个项目，一级架构已经够了。
- 你现在最该做的，不是继续争论技术方向，而是把“可编码级冻结件”补齐。
- 换句话说：
  - 大方向已经定了
  - 细节交付物还没完全冻结
- 如果你想看正式行动清单，权威入口是 `docs/architecture/implementation-freeze-checklist.md`。
- 如果你想按单条任务跟踪，权威入口是 `docs/architecture/issues/README.md`。

## 已经不差什么了
- 不差运行时主线：
  - 已经定了 [[.NET 10 LTS]]。
- 不差系统拆分：
  - 已经定了 [[微服务]] 拓扑、热路径、旁路和核心服务边界。
- 不差数据主线：
  - 已经定了 [[PostgreSQL]] 是事务真源，[[OpenSearch]] 只做聊天记录搜索读侧，[[Redis]] 只放临时状态，[[RabbitMQ]] 只管异步投递。
- 不差实时主线：
  - 已经定了实时聊天必须优先，AI、搜索、质检、统计都不能阻塞主聊天链路。
- 不差 AI 主线：
  - 已经定了 AI 必须经过 [[策略引擎]]、[[审计]]、[[租户隔离]] 和工具白名单。
- 不差恢复主线：
  - 已经定了断电恢复、重放、[[幂等]]、[[Outbox 模式]] 和自动恢复边界。
- 不差代码实现风格：
  - 已经定了 [[Pragmatic DDD]]、[[Workflow-first]]、[[CQRS]] 和 [[TDD]] 的组合方式。

## 现在真正还差什么
- 现在还差 3 类“开工冻结件”，另外 2 类已经完成。
- 已经完成的，是：
  - [[Contract Package V1]]
  - [[02-Domain/租户解析与授权冻结包 V1|租户解析与授权冻结包 V1]]
- 现在剩下的，不是讨论方向，而是为了让后续写代码时不跑偏。

- [x] 已冻结：[[Contract Package V1]]

- [x] 已冻结：operator/public 真实 [[HTTP 状态码语义]] + error envelope（错误信封）策略

你应该这样理解：
- 前 3 项是“已经能约束实现方向”的东西。
- 后 3 项是“不开工就会边写边猜”的东西。
- P1 包不是可以永远不写，而是可以在第一轮骨架后，按里程碑继续补齐。

## 已完成：接口与事件契约冻结包
- 这部分已经冻结成 [[Contract Package V1]]。
- 它已经把 `HTTP`、事件、错误 envelope、[[幂等]]、replay 规则固化到可评审、可联调、可做 contract test 的粒度。

现在你应该怎么理解它：
- 它不再是“将来要补”的清单，而是后续实现的正式入口。
- 只要公共接口、共享 payload、事件契约变了，就先改它，再改实现。
- operator/public 出错时要直接返回真实 [[HTTP 状态码语义]]，再用 `code` 和 [[Error Source]] 区分 gateway / system owner。
- `GET` surface 不接受 request body，也不允许 outer `200` 包住真实失败。

为什么重要：
- 微服务最容易失控的，不是代码本身，而是“契约漂移”。
- 现在这部分已经冻结，所以后续实现不需要再临场发明字段、错误码和 replay 规则。

## 还差：数据库详细设计冻结包
- 这里对应的正式冻结件叫 [[PostgreSQL Detailed Schema V1]]。
- 它不是只说“用 [[PostgreSQL]]”，而是要把表怎么建、索引怎么打、消息怎么重放、幂等怎么做，全部写清楚。

需要补齐：
- 每个核心服务的表结构
- 主键、唯一键、索引
- 时间分区
- `message_id`、`sequence`、幂等键
- 审计字段
- `outbox/inbox` 表设计
- 保留与归档策略

为什么重要：
- 真实生产里，很多问题不是“代码不会写”，而是“数据模型没定住”。
- 一旦表结构边写边想，后面重构成本会非常高。

## 已完成：租户、身份、权限冻结包
- 这部分已经冻结到 [[02-Domain/租户解析与授权冻结包 V1|租户解析与授权冻结包 V1]]。
- 现在已经明确：
  - 不同入口怎么做 [[Tenant Resolution]]
  - 服务端怎么构造 [[TenantContext]]
  - operator token 必须带哪些 claims
  - [[RBAC]] 和显式权限字符串怎么用
  - `platform_admin` 为什么必须走单独管理面

为什么重要：
- 很多多租户系统最后出事，不是因为数据库不行，而是因为租户解析和权限边界一开始没定死。
- 现在这部分已经定死，后续实现不该再靠聊天记忆猜 tenant、猜角色、猜管理员边界。

## 还差：外部集成冻结包
- 这个项目对应的正式冻结件叫 [[Upstream Integration Specs V1]]。
- 它不是纯内部系统，它要和 [[Webhook]]、企业微信、设备系统、订单系统真实打交道。
- 所以不能只说“后面再接”，要先把协议和失败语义定住。

需要补齐：
- 企业微信客服回调
- 公众号回调
- 上游重试规则
- 媒体回调与拉取
- 上游发消息的幂等方案
- 设备、订单、售后接口映射
- 部分失败后的补偿与对账策略

为什么重要：
- 外部系统最麻烦的点，不是“连上了没有”，而是“失败一次后怎么补、重复一次后会不会乱”。

## 还差：工程与发布冻结包
- 这部分对应的正式冻结件叫 [[Engineering Baseline V1]]。
- 这是把设计真正变成工程系统的那层。
- 你可以理解为“代码仓库、环境、发布流水线的统一规矩”。

需要补齐：
- 仓库目录和解决方案结构
- 共享库边界
- 本地开发拓扑
- 环境配置分层
- 持续集成流程
- [[Helm]] 约定
- `dev/staging/prod` 环境矩阵
- 关键服务的 [[SLO]] 与告警阈值

为什么重要：
- 如果没有这层，服务会越写越散，最后每个服务都有自己的小王国。

## 哪些可以边脚手架边补
- 下面这些不是完全不用做，而是可以在第一轮骨架后继续细化：
  - [[OpenSearch]] 搜索索引最终字段映射
  - [[ai-service]] 的配置字段级结构定义
  - [[media-service]] 的审核与素材治理流程细节
  - [[OpenTelemetry]]、[[SLO]]、故障演练脚本和 owner（责任人）模型

## 首个里程碑现在还卡在哪
- 第一个里程碑是 realtime conversation loop（实时会话闭环）。
- 其中已经基本冻结的有：
  - 入站消息 contract
  - 客服发送 contract
  - console 与 channel webhook 的租户解析
  - websocket replay 的 `message_id + sequence`
  - duplicate-delivery（重复投递）和 [[幂等]] 的合同级语义
- 现在最实质的剩余 blocker（阻塞项）是：
  - `message`
  - `conversation`
  - `outbox`
  - replay schema

为什么这点最关键：
- 因为接口和租户规则即使已经定了，落库规则没定，`conversation-service` 还是没法安全开工。

## 推荐开工顺序
1. 先冻结数据库详细设计。
2. 再冻结工程骨架、环境和发布规则。
3. 在做 [[channel-service]] 之前，冻结企业微信和设备系统集成。
4. 然后正式开始第一阶段编码：
   - [[conversation-service]]
   - [[routing-service]]
   - [[Realtime Gateway|realtime-gateway]]
   - [[channel-service]]
5. 再往后做：
   - [[search-service]]
   - [[media-service]]
   - [[knowledge-service]]
   - [[ai-service]]
   - [[analytics-service]]

## 你现在应该怎么学
- 把这篇当成“项目开工地图”。
- 学习时不要再问“要不要换大架构”，而要问：
  - 这个冻结件有没有写成正式文档？
  - 这个冻结件能不能直接指导建表、写接口、写测试？
  - 这个冻结件有没有明确边界和失败语义？

## 工作里怎么用
- 你在真实工作里，可以用这篇笔记去判断一个项目是不是“能开工”。
- 只要发现项目还在反复讨论技术选型，却没有契约、表结构、权限、集成、工程基线，那就说明它离稳定开工还差一层。

## 面试怎么讲
- 你可以这样说：
  “我会把架构完成度分成两层。第一层是蓝图级，决定技术方向和服务边界。第二层是可编码级，冻结契约、数据库、权限、外部集成和工程基线。只有第二层补齐，团队实现才不会跑偏。”
