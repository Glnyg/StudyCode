# 公共接口与事件目录

对应正式文档：`docs/api/public-contracts-and-events.md`

## 这是什么
- 这篇是系统对外和对内沟通的“语言规范”。
- `HTTP/gRPC` 是同步接口。
- 事件是异步通信。

## 核心事件
- [[MessageAppended]]
- [[MessageRedacted]]
- [[ConversationAssigned]]
- [[TransferCompleted]]
- [[AiHandoffChanged]]
- [[AssetSent]]
- [[UrgentInterventionTriggered]]
- [[UrgentInterventionAcknowledged]]
- [[ResponseTimeoutAlertTriggered]]
- [[ResponseTimeoutAlertCleared]]
- [[ManagementNotificationDispatched]]

## 核心规则
- 事件必须带 `tenant_id`
- 事件必须可 [[幂等]] 消费
- 改事件契约要改 ADR
- 管理通知事件现在还要能区分高风险告警和 [[响应时限告警]] 来源

## 为什么重要
- 很多[[微服务]]系统不是业务逻辑崩，而是“契约漂移”崩。
- 如果字段今天加、明天改、后天删，但没人记，系统很快会变得不可维护。

## 在本项目里怎么用
- 对外的 `HTTP/gRPC` 接口由共享契约文档定义，请求、响应、错误码和幂等约束都要先写进正式文档。
- 对内事件至少要带 `tenant_id`、[[审计]] 上下文和可重放所需字段。
- 一旦涉及告警、AI 或搜索的边界字段变更，文档、Obsidian 教学镜像和 ADR 要一起更新。

## 工作里怎么用
- 改接口前先改文档。
- 改事件前先想兼容性。
- 对下游影响大的字段改动要补迁移说明。
- 涉及企微、飞书这种内部通知时，也要把告警事件、`source_type/source_id` 和补偿语义写清楚。

## 面试怎么说
- “我会把 API 和事件契约当成正式资产管理，而不是聊天里口头约定。这样做能减少微服务协作里的隐形故障。”
