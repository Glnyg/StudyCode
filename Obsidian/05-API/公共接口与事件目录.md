# 公共接口与事件目录

对应正式文档：`docs/api/public-contracts-and-events.md`

## 这是什么
- 这篇笔记是 [[Contract Package V1]] 的学习入口。
- 它解释这个项目如何把 `HTTP` 接口、事件、错误码、幂等和 replay 规则变成正式资产。

## 现在的正式入口
- 总入口：`docs/api/public-contracts-and-events.md`
- 合同冻结包：`docs/api/contract-package-v1/README.md`
- 机器可读合同主要包括：
  - [[OpenAPI]]
  - [[JSON Schema]]
- 人类可读说明默认中文优先，但 service、event、schema key、path 等正式标识保持英文。

## 这套包里有什么
- `[[conversation-service]]` 和 [[Realtime Gateway|realtime-gateway]] 的会话查询、消息回放、发送与重放边界
- `[[routing-service]]` 的队列、分配、转接、在线状态、紧急介入和[[响应时限告警]]合同
- `[[search-service]]` 的搜索、补全、健康检查和 `search_after` 规则
- `[[media-service]]` 的资产上传、审核、列表和预览 URL 合同
- `[[ai-service]]` 的建议回复、执行、策略快照、审计和 tool gating 合同

## 核心规则
- trusted `tenant_id` 不能靠客户端自己声明，必须来自可信解析上下文
- 所有 side-effecting `POST` 都要明确是否要求 [[幂等]] `Idempotency-Key`
- 版本化配置写入要用 `If-Match`
- 搜索分页要用 `search_after`
- 实时重放要用 `last_seen_sequence`
- 事件必须带租户、关联、版本和可重放所需字段

## 为什么重要
- 很多[[微服务]]系统真正难的是“边界是否一致”，不是某一个方法会不会写。
- 这套合同包的价值，是让后续实现者不用再猜：
  - header 怎么传
  - 错误码怎么命名
  - replay 怎么定义
  - breaking change 什么时候要升格到 ADR

## 在本项目里怎么用
- 先改正式合同，再改实现。
- 一旦改公共接口、事件、共享 payload，就要同步正式文档、Obsidian 教学镜像和相关 ADR。
- 如果你发现某个服务实现与合同不一致，默认以正式合同为准，而不是以“现有代码可能就是这样”为准。

## 工作里怎么用
- 要做并行开发时，先看合同包，而不是先看各自脑补的 DTO。
- 要写 contract test 时，直接用 schema 和 example 做校验。
- 要做 review 时，先问“有没有破坏合同”，再问“代码写得漂不漂亮”。

## 面试怎么说
- “我会把公共接口和事件做成正式合同包，里面同时冻结 OpenAPI、JSON Schema、错误 envelope、幂等和 replay 规则。这样多服务和多 agent 协作时，不会因为契约漂移反复返工。”
