# 测试策略与质量闸门

对应正式文档：`docs/testing/test-strategy-and-quality-gates.md`

## 这是什么
- 这篇定义“tests have guarantees”在这个仓库里到底是什么意思。
- 核心不是测试数量，而是改动有没有被正确的测试层保护。

## 为什么重要
- 一个通过的测试套件，如果没有覆盖最危险的那一层，仍然可能把缺陷放进生产。
- 对这个项目来说，多租户、重放、AI 策略、搜索重建和公共契约都不能只靠低价值 smoke test。

## 哪类改动需要哪类测试
- 核心域改动：领域不变量测试 + 至少一条应用服务或集成路径
- 编排/工作流改动：应用服务测试 + 集成/契约测试 + 重试或幂等覆盖
- 搜索改动：投影测试 + 查询行为测试 + 重建或重放覆盖
- AI 策略改动：策略决策测试 + 回退测试 + 审计完整性测试
- 公共契约改动：契约测试 + 文档里的兼容性说明

## 在本项目里怎么用
- 改 `conversation-service`、`routing-service` 这类真源服务时，先确认有没有领域不变量测试保护核心规则。
- 改搜索、AI、告警、Webhook 或重放逻辑时，优先补能体现失败语义的集成或契约测试。
- 如果只是加了一个 controller 或 smoke test，但最危险的业务层没被覆盖，这里会直接判定为不够。

## 工作里怎么用
- 每次说“这个改动测过了”之前，先回答两个问题：
  - 这个改动最危险的层是哪一层
  - 我现在的测试是不是真的覆盖了那一层
- 代码评审时，不要只看有没有测试文件，要看测试能否在改动出错时真正失败。

## 面试怎么说
- “我会把测试策略按风险和层次来设计。核心域守不变量，工作流看集成和幂等，搜索看投影和重建，AI 看策略和审计，契约改动看兼容性。”
