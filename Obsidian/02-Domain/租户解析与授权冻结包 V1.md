# 租户解析与授权冻结包 V1

对应正式文档：`docs/domain/tenant-resolution-and-authorization-v1.md`

## 这是什么
- 这篇笔记把 [[Tenant Resolution]]（租户解析）和授权规则讲成可直接开工的冻结包。
- 它回答的不是“要不要做 [[多租户]]”，而是：
  - tenant 到底从哪来
  - `Authorization` token 里必须带什么
  - 谁能看、谁能写、谁能转接、谁能发 AI 指令
  - `platform_admin` 为什么不能直接走普通坐席接口

## 你先记住
- 所有 operator request（操作员请求）的租户，都只能通过 [[Tenant Resolution]] 变成服务端可信的 [[TenantContext]]。
- operator token 是单租户 session token（单租户会话令牌），不是“一个 token 同时切很多租户”的随意模型。
- 权限判断看的是显式 `permissions`，不是前端按钮有没有显示。
- `platform_admin` 必须走显式 admin surface（管理面），不能假装成普通租户坐席。
- tenant / permission / admin boundary 拒绝时，要直接返回真实 [[HTTP 状态码语义]]，不能用 outer `200` 把失败藏起来。
- `gateway` 先拦下来的问题，和服务自己判定出来的问题，要通过 [[Error Source]] 和 `code` 明确区分。

## 为什么重要
- 多租户系统真正容易出事故的地方，常常不是数据库，而是“tenant 到底是谁”这一步。
- 只要 tenant 解析模糊、权限判断靠猜、管理员边界不清，后面 [[search-service]]、[[ai-service]]、[[media-service]]、[[routing-service]] 都会埋串租户风险。
- 所以这份冻结包本质上是在给 [[租户隔离]]、[[RBAC]] 和 [[审计]] 打地基。

## 在本项目里怎么用
- operator console：
  - bearer token 解析出单个 active `tenant_id`
  - 服务端把它变成 [[TenantContext]]
  - 再根据 `permissions` 做授权
- Enterprise WeChat webhook：
  - 先验签
  - 再查 verified channel binding（已验证渠道绑定）
  - 最后才能得到 [[TenantContext]]
- internal event：
  - 必须同时有 `tenant_id` 和 trusted producer（可信生产者）
  - 否则不允许产生业务副作用
- `platform_admin`：
  - 必须显式指定目标租户
  - 必须带审计原因
  - 必须走单独 admin surface
- operator/public `GET`：
  - 不接受 request body
  - 如果客户端硬发 body，要直接拒绝，而不是静默继续

## 这轮具体冻结了什么
- trusted resolution paths（可信解析路径）：
  - operator token
  - verified channel binding
  - trusted internal event
- operator token claims：
  - `sub`
  - `tenant_id`
  - `actor_type`
  - `role`
  - `permissions`
  - `session_id`
- canonical permissions（规范权限）：
  - `conversation.read`
  - `conversation.reply`
  - `conversation.assign`
  - `conversation.transfer`
  - `search.read`
  - `media.review`
  - `ai.policy.publish`
  - `tenant.config.write`
- role baseline（角色基线）：
  - `agent`
  - `supervisor`
  - `qa`
  - `tenant_admin`

## 工作里怎么落地
- 评审需求时，先问：
  - 这个入口怎么做 [[Tenant Resolution]]？
  - 结果有没有落成 [[TenantContext]]？
  - 授权判断到底看什么 permission？
  - cross-tenant 失败时是 `403` 还是 `404`？
- 评审实现时，重点抓：
  - 有没有客户端自报 `tenant_id`
  - 有没有拿角色名直接代替权限判断
  - 有没有让 `platform_admin` 走普通 operator API
  - 有没有把 auth / tenant / permission 失败包装成 outer `200`
  - 有没有让 `GET` request body 进入业务语义
  - privileged write（特权写）有没有带 [[审计]] 所需字段

## 面试里怎么表达
- “我会把多租户的安全边界拆成两层：先做 trusted tenant resolution，再做 permission-based authorization。这样 token、webhook、internal event 都有唯一可信来源，管理员边界也不会和普通租户接口混在一起。”
